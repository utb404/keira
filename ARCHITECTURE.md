## Введение

Библиотека представляет собой интеллектуальную систему для автоматической генерации автотестов UI на основе тест-кейсов с применением технологий Large Language Models (LLM). Библиотека спроектирована для создания качественных, поддерживаемых и соответствующую корпоративным стандартам автотестов с использованием паттерна Page Object Model и лучших практик автоматизации тестирования.

### Назначение

Библиотека решает проблему трудоемкого и рутинного процесса создания автотестов UI, автоматизируя генерацию кода тестов на основе структурированных тест-кейсов в формате JSON. Применение LLM позволяет создавать контекстуально правильные тесты, учитывающие структуру существующего проекта и корпоративные стандарты разработки.

## Миссия и цели

### Миссия

Генерация автотестов UI, соответствующих корпоративным шаблонам с применением лучших практик по разработке автотестов.

### Основные цели

1. **Автоматизация генерации тестов**: Минимизация ручного труда при создании автотестов UI
2. **Соблюдение стандартов**: Генерация тестов, соответствующих корпоративным шаблонам и паттернам
3. **Качество кода**: Применение лучших практик автоматизации тестирования
4. **Адаптивность**: Анализ существующих проектов и адаптация под их структуру
5. **Расширяемость**: Возможность интеграции с различными инструментами и технологиями

## Архитектура системы

### Общая архитектура

Библиотека построена по модульному принципу с четким разделением ответственности между компонентами:

```
┌─────────────────────────────────────────────────────────────┐
│                    Генератор Автотестов                     │
│                      (Main Entry Point)                      │
└───────────────────────┬─────────────────────────────────────┘
                        │
        ┌───────────────┼───────────────┐
        │               │               │
┌───────▼──────┐ ┌──────▼──────┐ ┌──────▼──────┐
│  Repository  │ │   Test Case │ │    LLM      │
│   Analyzer   │ │   Parser    │ │  Provider   │
└───────┬──────┘ └──────┬──────┘ └──────┬──────┘
        │               │               │
        └───────────────┼───────────────┘
                        │
        ┌───────────────┼───────────────┐
        │               │               │
┌───────▼──────┐ ┌──────▼──────┐ ┌──────▼──────┐
│  Template    │ │   Code      │ │   Output    │
│  Generator   │ │  Generator  │ │   Manager   │
└──────────────┘ └─────────────┘ └─────────────┘
```

### Уровни архитектуры

1. **Уровень ввода данных** (Input Layer)
   - Парсинг тест-кейсов из JSON
   - Валидация структуры тест-кейсов
   - Загрузка конфигураций

2. **Уровень анализа** (Analysis Layer)
   - Индексация репозитория
   - Анализ структуры проекта
   - Извлечение паттернов и шаблонов

3. **Уровень генерации** (Generation Layer)
   - Формирование промптов для LLM
   - Генерация кода через LLM
   - Валидация сгенерированного кода

4. **Уровень вывода** (Output Layer)
   - Форматирование кода
   - Создание файловой структуры
   - Генерация метаданных

## Основные компоненты

### 1. Repository Analyzer (Анализатор репозитория)

**Назначение**: Подключение, индексация и анализ репозитория с автотестами GitLab для извлечения шаблонов и паттернов.

**Основные функции**:
- Подключение к GitLab через API или Git-протокол
- Полная индексация структуры репозитория
- Анализ существующих тестов и извлечение паттернов
- Определение корпоративных шаблонов (структура проекта, именование, стиль кода)
- Выявление использования библиотеки qautils
- Анализ применения паттерна Page Object Model
- Извлечение конфигураций и настроек

**Ключевые классы**:
- `RepositoryConnector` - управление подключением к GitLab
- `RepositoryIndexer` - индексация файлов и структуры
- `PatternExtractor` - извлечение паттернов и шаблонов
- `TemplateAnalyzer` - анализ корпоративных шаблонов

**Стратегия индексации**:
1. **Первичная индексация**: Выполняется единожды при первом запуске
   - Клонирование/подключение к репозиторию
   - Полное сканирование структуры проекта
   - Извлечение всех тестовых файлов
   - Анализ зависимостей и импортов
   - Создание индекса метаданных

2. **Обновление индекса**: Выполняется при необходимости вручную
   - Проверка изменений в репозитории
   - Инкрементальное обновление индекса
   - Переанализ измененных файлов

3. **Хранение индекса**:
   - Локальное кэширование индекса в структурированном формате (JSON/YAML)
   - Возможность экспорта/импорта индекса
   - Версионирование индекса

### 2. Test Case Parser (Парсер тест-кейсов)

**Назначение**: Парсинг и валидация тест-кейсов в формате JSON.

**Основные функции**:
- Парсинг JSON-структуры тест-кейса
- Валидация обязательных полей
- Извлечение шагов теста
- Нормализация данных для генерации
- Поддержка множественных тест-кейсов

**Структура тест-кейса** (на основе примера):
```json
{
  "id": "PLAYWRIGHT_INTRO_001",
  "name": "[E2E] Playwright Python. Проверка установки и примера теста",
  "description": "...",
  "preconditions": "...",
  "expectedResult": "...",
  "epic": "...",
  "feature": "...",
  "story": "...",
  "component": "...",
  "testLayer": "E2E",
  "severity": "NORMAL",
  "priority": "1",
  "environment": "Chrome Headless",
  "browser": "chromium",
  "tags": "playwright, pytest, e2e",
  "testType": "automated",
  "steps": [...]
}
```

**Ключевые классы**:
- `TestCaseParser` - основной парсер JSON
- `TestCaseValidator` - валидация структуры
- `TestCaseNormalizer` - нормализация данных
- `StepExtractor` - извлечение и обработка шагов

### 3. LLM Provider (Провайдер LLM)

**Назначение**: Интеграция с Ollama для генерации кода автотестов.

**Основные функции**:
- Подключение к Ollama (локально и удаленно)
- Выбор и управление моделями
- Формирование контекстных промптов
- Генерация кода с учетом шаблонов
- Обработка ответов LLM

**Поддерживаемые режимы**:
- **Локальный режим**: Подключение к локальному экземпляру Ollama
- **Удаленный режим**: Подключение к удаленному серверу Ollama
- **Выбор модели**: Поддержка различных моделей (llama2, codellama, mistral и др.)

**Ключевые классы**:
- `OllamaProvider` - основной класс для работы с Ollama
- `PromptBuilder` - построитель промптов
- `ModelManager` - управление моделями
- `ResponseProcessor` - обработка ответов LLM

**Структура промптов**:
1. **Системный промпт**: Определяет роль и контекст
2. **Контекстный промпт**: Включает информацию о проекте и шаблонах
3. **Задачный промпт**: Конкретная задача генерации теста
4. **Примеры**: Примеры из проанализированного репозитория

### 4. Template Generator (Генератор шаблонов)

**Назначение**: Формирование шаблонов для генерации тестов на основе анализа репозитория.

**Основные функции**:
- Извлечение структуры проекта из индекса
- Определение паттернов именования
- Выявление используемых паттернов Page Object
- Формирование шаблонов конфигураций
- Создание шаблонов фикстур pytest
- Генерация шаблонов страниц (Page Objects)

**Извлекаемые шаблоны**:
- Структура директорий (tests/, pages/, conftest.py)
- Именование файлов и классов
- Структура Page Object классов
- Паттерны использования qautils
- Стиль кода и комментариев
- Структура Allure-декораторов
- Конфигурации браузера

**Ключевые классы**:
- `TemplateExtractor` - извлечение шаблонов
- `TemplateBuilder` - построитель шаблонов
- `PatternMatcher` - сопоставление паттернов

### 5. Code Generator (Генератор кода)

**Назначение**: Генерация кода автотестов на основе тест-кейсов, шаблонов и LLM.

**Основные функции**:
- Генерация Page Object классов
- Генерация тестовых методов
- Генерация фикстур pytest
- Интеграция с qautils
- Применение паттернов из репозитория
- Генерация Allure-декораторов

**Процесс генерации**:
1. Анализ шагов тест-кейса
2. Определение необходимых Page Objects
3. Генерация селекторов и взаимодействий
4. Создание тестового метода
5. Добавление проверок (assertions)
6. Интеграция с Allure Report

**Ключевые классы**:
- `PageObjectGenerator` - генерация Page Object классов
- `TestMethodGenerator` - генерация тестовых методов
- `FixtureGenerator` - генерация фикстур
- `CodeFormatter` - форматирование кода

### 6. CDP Integration (Интеграция с Chrome DevTools Protocol)

**Назначение**: Использование CDP для более точного определения селекторов и взаимодействий с браузером.

**Основные функции**:
- Подключение к браузеру через CDP
- Анализ DOM-структуры
- Автоматическое определение селекторов
- Извлечение метаданных элементов
- Валидация селекторов
- Генерация стабильных локаторов

**Применение**:
- Определение оптимальных селекторов для элементов
- Валидация сгенерированных локаторов
- Анализ доступности элементов
- Извлечение информации о структуре страницы

**Ключевые классы**:
- `CDPClient` - клиент для работы с CDP
- `SelectorExtractor` - извлечение селекторов
- `ElementAnalyzer` - анализ элементов через CDP

### 7. Output Manager (Менеджер вывода)

**Назначение**: Управление выводом сгенерированных файлов и создание структуры проекта.

**Основные функции**:
- Создание структуры директорий
- Генерация файлов тестов
- Генерация Page Object файлов
- Обновление существующих файлов
- Резервное копирование
- Генерация метаданных

**Ключевые классы**:
- `FileManager` - управление файлами
- `DirectoryBuilder` - построение структуры директорий
- `CodeWriter` - запись кода в файлы

## Потоки данных

### Основной поток генерации теста

```
1. Загрузка тест-кейса (JSON)
   │
   ├─► Парсинг и валидация тест-кейса
   │
2. Подключение к репозиторию (если необходимо)
   │
   ├─► Проверка наличия индекса
   │   ├─► Индекс существует → Загрузка
   │   └─► Индекс отсутствует → Индексация репозитория
   │
3. Анализ шагов тест-кейса
   │
   ├─► Определение необходимых Page Objects
   │
4. Генерация промпта для LLM
   │
   ├─► Включение контекста проекта
   ├─► Включение шаблонов из репозитория
   ├─► Включение описания тест-кейса
   │
5. Генерация кода через LLM
   │
   ├─► Генерация Page Object классов
   ├─► Генерация тестового метода
   │
6. Валидация и форматирование кода
   │
7. Создание файловой структуры
   │
   └─► Запись файлов в проект
```

### Поток индексации репозитория

```
1. Подключение к GitLab репозиторию
   │
   ├─► Аутентификация (токен/SSH)
   │
2. Клонирование/сканирование репозитория
   │
3. Индексация структуры
   │
   ├─► Поиск тестовых файлов
   ├─► Поиск Page Object классов
   ├─► Поиск фикстур
   ├─► Анализ импортов и зависимостей
   │
4. Извлечение паттернов
   │
   ├─► Структура директорий
   ├─► Именование
   ├─► Использование qautils
   ├─► Паттерны Page Object
   │
5. Создание шаблонов
   │
6. Сохранение индекса
   │
   └─► Кэширование в локальном хранилище
```

## Интеграции

### 1. GitLab

**Назначение**: Подключение к репозиторию с автотестами для анализа.

**Методы подключения**:
- GitLab API (REST API)
- Git протокол (HTTPS/SSH)

**Используемые библиотеки**:
- `python-gitlab` - для работы с GitLab API
- `gitpython` - для работы с Git репозиториями

**Функциональность**:
- Аутентификация через токены
- Клонирование репозиториев
- Анализ истории коммитов
- Извлечение файлов и структуры

### 2. Ollama

**Назначение**: Генерация кода через LLM.

**Режимы работы**:
- Локальный: `http://localhost:11434`
- Удаленный: настраиваемый URL

**Используемые библиотеки**:
- `ollama` - официальный Python клиент для Ollama

**Функциональность**:
- Выбор модели
- Управление контекстом
- Генерация кода
- Обработка ответов

### 3. qautils

**Назначение**: Использование возможностей библиотеки qautils в генерируемых тестах.

**Применение**:
- `gpn_qa_utils.ui.browser_launcher.BrowserLauncher` - инициализация браузера
- `gpn_qa_utils.ui.pages.base.BasePage` - базовый класс страницы
- `gpn_qa_utils.ui.page_factory.*` - компоненты для взаимодействия
- Интеграция с Allure Report

**Паттерны использования**:
- Создание фикстур с BrowserLauncher
- Наследование от BasePage
- Использование компонентов (Button, Input, Link и др.)
- Применение Allure-декораторов

### 4. Playwright

**Назначение**: Основа для UI-автотестов.

**Применение**:
- Управление браузером
- Взаимодействие с элементами
- Навигация по страницам

### 5. Allure Report

**Назначение**: Генерация отчетов о тестировании.

**Применение**:
- Декораторы для тестов (@allure.epic, @allure.feature, @allure.story)
- Шаги тестов (allure.step)
- Прикрепление скриншотов и метаданных

### 6. Chrome DevTools Protocol (CDP)

**Назначение**: Дополнительный анализ браузера для точной генерации селекторов.

**Применение**:
- Подключение к браузеру через CDP
- Анализ DOM
- Извлечение селекторов
- Валидация элементов

**Используемые библиотеки**:
- `playwright` (встроенная поддержка CDP)
- `pychrome` (опционально для прямого доступа)

## Принципы проектирования

### 1. Модульность

Библиотека построена по принципу модульной архитектуры, где каждый модуль отвечает за определенную функциональность. Модули слабо связаны и могут развиваться независимо.

### 2. Расширяемость

Архитектура позволяет легко добавлять новые:
- Провайдеры LLM (не только Ollama)
- Источники тест-кейсов (не только JSON)
- Генераторы кода для различных фреймворков
- Анализаторы репозиториев

### 3. Конфигурируемость

Все параметры работы библиотеки настраиваются через конфигурационные файлы:
- Путь к репозиторию
- Настройки Ollama

## Формат конфигурации

### Основной конфигурационный файл (config.yaml)

```yaml
# Настройки репозитория
repository:
  type: "gitlab"  # gitlab, git
  url: "https://gitlab.example.com/project/repo.git"
  auth:
    type: "token"  # token, ssh
    token: "${GITLAB_TOKEN}"  # из переменных окружения
  index_path: "./.test_generator/index.json"
  auto_index: false  # автоматическая индексация при изменениях

# Настройки LLM
llm:
  provider: "ollama"
  ollama:
    url: "http://localhost:11434"  # локальный или удаленный
    model: "codellama:13b"
    temperature: 0.3
    max_tokens: 4000
    timeout: 300

# Настройки генерации
generation:
  output_path: "./generated_tests"
  overwrite_existing: false
  format_code: true
  validate_code: true
  
# Настройки CDP (опционально)
cdp:
  enabled: false
  browser: "chromium"
  headless: true
  
# Настройки шаблонов
templates:
  page_object:
    base_class: "BasePage"
    import_path: "gpn_qa_utils.ui.pages.base"
  fixtures:
    browser_launcher: "BrowserLauncher"
    import_path: "gpn_qa_utils.ui.browser_launcher"
    
# Логирование
logging:
  level: "INFO"
  file: "./test_generator.log"
  format: "%(asctime)s - %(name)s - %(levelname)s - %(message)s"
```

            strategy="by_role",

            value="Getting Started",
            allure_name="Ссылка Getting Started"
        )
